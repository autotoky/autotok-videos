### Fix API obtener_lote — Skip lotes completados

**Problema**: `GET /api/lotes?cuenta=X` (sin fecha) devolvía siempre el lote más viejo con `fecha >= hoy`, incluso si todos sus videos ya tenían resultado. El lote del día 8 (7 videos, 7 con resultado = 0 pendientes) bloqueaba al día 9, causando que PUBLICAR.bat en el PC de la operadora mostrara "No hay videos pendientes".

**Causa raíz**: Query SQL `ORDER BY fecha ASC LIMIT 1` no tenía en cuenta si el lote estaba completado.

**Fix aplicado** en `autotok-api/api/lotes.py`:
- Eliminado `LIMIT 1` del query
- Añadido loop que itera lotes por fecha ascendente
- Para cada lote: consulta resultados existentes y cuenta pendientes (videos sin resultado)
- Si `pendientes > 0`: devuelve ese lote
- Si todos los videos tienen resultado: salta al siguiente lote
- Si búsqueda por `?fecha=X`: devuelve siempre (sin skip)
- Commit: `fix: skip completed lotes in GET /api/lotes` — pushed a GitHub, desplegado en Vercel

### QUA-155 — Migración completa a Turso (fuente de verdad)

**Problema**: El sistema tenía dos bases de datos desconectadas: `autotok.db` local (SQLite, 11 tablas) y Turso cloud (4 tablas: lotes, resultados, descartes, version). Los resultados de publicación de operadoras nunca volvían a la BD local de Sara. Esto causaba inconsistencias constantes.

**Solución**: Migrar TODO a Turso como fuente única de verdad, usando embedded replica para lecturas locales rápidas.

**Pasos completados**:

1. **Schema migrado a Turso**: 11 tablas + 14 índices creados en Turso cloud via HTTP API pipeline.

2. **Datos migrados** (todas las tablas al 100%):
   - productos: 27, producto_bofs: 29, variantes_overlay_seo: 161
   - audios: 73, material: 779, videos: 1618
   - hook_variante_usado: 990, combinaciones_usadas: 1462
   - historial_programacion: 18, video_publish_log: 92
   - cuentas_config: 0 (vacía)

3. **FK constraint issue resuelto**: 440 videos referenciaban 27 audio_ids huérfanos (audios eliminados). Se migraron con FK checks desactivado en la misma pipeline.

4. **db_config.py v4.1** — Turso HTTP API (zero deps extra):
   - **v4.0** usaba `libsql-experimental` (embedded replica) pero requiere compilar Rust → falla en Windows
   - **v4.1** usa Turso HTTP API directamente (`urllib` — incluido en Python, zero installs)
   - Si `turso_config.json` existe → usa Turso HTTP API (queries van a la nube)
   - Si no → cae a SQLite local (backward compatible para operadoras)
   - `TursoHTTPCursor` y `TursoHTTPConnection` emulan sqlite3 cursor/connection
   - `DictRow` emula `sqlite3.Row` (acceso por nombre `row['col']` y por índice `row[0]`)
   - 0 cambios necesarios en el resto del codebase
   - Cada query individual ~300ms (HTTP a Europa), operaciones completas ~1-2s
   - Funciones: `is_turso_enabled()`, `sync_replica()` (no-op en HTTP mode)

5. **turso_config.json** creado (en .gitignore por contener token):
   ```json
   {
     "sync_url": "libsql://autotok-autotok.aws-eu-west-1.turso.io",
     "auth_token": "...",
     "local_replica": "autotok_replica.db"
   }
   ```

6. **Verificación**: Todas las queries reales del sistema probadas con HTTP API:
