### Fase B: Documentacion lock files

**Problema:** La VM crea archivos `.git/index.lock` al hacer `git add`/`git commit` pero no puede borrarlos (operacion `unlink` bloqueada en el filesystem montado). El lock queda permanente y bloquea operaciones git futuras.

**Solucion documentada en INSTRUCCIONES_CLAUDE.md:**
- Para `autotok-api`: normalmente funciona si no hay lock previo
- Para `autotok-videos`: NUNCA hacer git add/commit desde la VM. Sara lo hace desde PowerShell
- Comando para limpiar locks: `Get-ChildItem -Path ".git" -Recurse -Filter "*.lock" | Remove-Item -Force`

### Fase I: QUA-303 — Filtro origen con desglose de datos

**Problema:** El filtro Origen en Rendimiento Producto tenía opciones Todos/Nuestro/Externo/Mixto y solo filtraba filas por tipo de producto. Sara necesitaba que al seleccionar "Externo" mostrara solo los datos (ventas, engagement, GMV) provenientes de videos externos, y viceversa para "Nuestro".

**Cambios implementados:**

1. **Backend (`analytics.py` — `_data_producto()`):**
   - Query de ventas: agregados condicionales con sufijo `_n` (nuestro) y `_e` (externo) para ventas, unidades, gmv, comisiones
   - Query de engagement: campo `is_ext` (0/1) en subquery UNION + agregados condicionales para views, likes, comments, shares, saves por origen
   - Cada fila incluye campos split: views_n/views_e, gmv_n/gmv_e, eng_rate_n/eng_rate_e, etc.

2. **Frontend (Producto view):**
   - Filtro cambiado de "Todos/Nuestro/Externo/Mixto" a "Todos/Nuestro/Externo"
   - `getFiltered()`: cuando se selecciona nuestro/externo, remapea campos de datos a sus versiones `_n` o `_e` y filtra productos sin videos de ese origen
   - Columna Videos muestra conteo según filtro activo
   - Indicador "(solo nuestros)" / "(solo externos)" en contador

**Commit:** `d030ff0` en autotok-api.

### Fase J: QUA-293 — Convención formato números y fechas

**Problema:** El dashboard usaba formatos inconsistentes para números, monedas, porcentajes y fechas.

**Convención implementada:**
- Números <1.000: formato `XXX,xx` (2 decimales, coma decimal)
- Números ≥1.000: sin decimales, punto separador de miles
- Moneda: siempre con símbolo `€`
- Porcentajes: siempre con símbolo `%`
- Fechas: DD/MM/AAAA
- Horas: 24h

**Cambios:**
1. **analytics.py**: nueva función `fmtVal()` con la convención, `fmtEur()` usa `fmtVal() + ' €'`, `fmtDate()` global para DD/MM/YYYY, chart labels corregidos
2. **productos.py**: añadidas `fmtVal`, `fmtEur`, `fmtDate`. GMV y fecha creación formateados
3. **ventas.py**: GMV sigue la convención
4. **estado.py**: ya estaba conforme

**Commit:** `43a4580` en autotok-api.

### Fase K: Fix sort double-cycling en analytics

**Problema:** Al hacer click en una columna para ordenar en Rendimiento Producto/Video, el sort avanzaba dos estados en vez de uno. Esto hacía que pareciera que no funcionaba (especialmente GMV).

**Causa:** `doSort()` llamaba a `sortTable()` que avanzaba `sortCycle`, y luego `renderTable()` volvía a llamar `sortTable()` avanzándolo otra vez.

**Fix:** Separada la lógica en `advanceSort()` (cambia estado) y `sortTable()` (solo ordena). `doSort` llama a `advanceSort` + `renderTable`.

**Commit:** `7ae3fe5` en autotok-api.
