## QUA-229: Fix hora/fecha inline editing en dashboard Calendario

### Problema

Sara reportó que al editar hora o fecha de un video en el dashboard de estado (Calendario), el cambio no se guardaba — volvía al valor anterior. El backend (API) funcionaba correctamente (verificado con curl), el bug era puramente frontend.

### Root cause

Las funciones `editTime()`, `editDate()` y `saveSchedule()` en `api/estado.py` usaban `VIDEOS[gidx]` para acceder al video, pero `gidx` es un índice en el array **filtrado/ordenado** devuelto por `getFiltered()`, NO en el array original `VIDEOS`. Cuando la tabla estaba ordenada o filtrada, se editaba el video incorrecto (o ninguno).

La función `onEstadoChange()` ya usaba `getFiltered()[gidx]` correctamente — las funciones de edición no.

### Fix

Cambio en 3 funciones de `api/estado.py`:
- `editDate()` (línea ~1413): `VIDEOS[gidx]` → `getFiltered()[gidx]`
- `editTime()` (línea ~1426): `VIDEOS[gidx]` → `getFiltered()[gidx]`
- `saveSchedule()` (línea ~1439): `VIDEOS[gidx]` → `getFiltered()[gidx]`

## QUA-244: Sincronización panel ↔ lotes (SIEMPRE mismos datos)

### Problema

Tras editar hora/fecha en el panel, el lote en Turso se actualizaba correctamente (via `_sincronizar_lote()`), pero los JSON locales en `SynologyDrive/{cuenta}/_lotes/` quedaban desfasados. Sara exigió: "SIEMPRE que te digo que un cambio en el panel tiene q estar en todos sitios me refiero a TODOS. Es decir, se publique desde donde se publique debe hacerse con los datos del panel."

### Análisis

El flujo de lotes tiene 3 capas:
1. **Turso (tabla `lotes`)** — actualizado por `_sincronizar_lote()` en cada cambio del panel ✅
2. **API (`/api/lotes`)** — lee de Turso, cross-check con tabla `videos`, siempre fresco ✅
3. **JSON local** (`SynologyDrive/{cuenta}/_lotes/`) — snapshot estático, se desfasa ⚠️

`publicar_facil.py` (PUBLICAR.bat) ya tiene comportamiento API-first: busca lotes en la API primero y sobreescribe el JSON local con lo que devuelve la API (línea ~127). Solo usa JSON local como fallback si la API no está disponible. Es decir, cuando la operadora ejecuta PUBLICAR.bat, siempre obtiene datos frescos de la API.

### Solución

1. **No es necesario borrar los JSON locales** — sirven como cache/fallback
2. **Creado `scripts/sync_lotes.py`** — script para sincronizar manualmente los JSON locales desde la API de Turso. Funciones: `sync_cuenta(cuenta, drive_path, fecha=None)` para uso programático, y `main()` con argumentos `--cuenta` y `--fecha` para uso CLI
3. **Creado `SINCRONIZAR.bat`** — wrapper que ejecuta `sync_lotes.py` para que Sara pueda refrescar los JSON locales con doble-click si quiere verificar visualmente

### Archivos creados

- `SynologyDrive/kevin/scripts/sync_lotes.py` — sincronizador de lotes API → JSON local
- `SynologyDrive/kevin/SINCRONIZAR.bat` — wrapper para ejecutar sync_lotes.py

### Decisiones importantes

1. **API es la fuente de verdad para lotes** — JSON locales son cache, no fuente primaria
2. **publicar_facil.py ya lee de API primero** — la operadora siempre publica con datos del panel
3. **`_sincronizar_lote()` cubre todos los cambios del panel** — hora, fecha, estado, es_ia
4. **No se necesita sincronización automática de JSON locales** — se refrescan cuando PUBLICAR.bat los lee de la API

## QUA-231: Programador pone fechas del pasado (overnight window bug)

### Problema

Sara lanzó el programador web a las 07:10 con lotopdevicky (ventana 06:00-02:00). Un video se asignó a las 01:15 del 13/03, que es 6 horas en el pasado. No era un tema de buffer (15 vs 30 min).

### Root cause

Con ventana overnight (06:00-02:00), `_generar_horario()` genera slots hasta las 02:00 del día siguiente. Internamente trabaja en minutos (360-1560). Un slot a las 01:15 (minuto 1515) se convierte a "01:15" haciendo `h -= 24`. Pero al asignar la fecha, usaba siempre `fecha_str` (13/03) sin considerar que las 01:15 son del DÍA SIGUIENTE (14/03).

### Fix

- Detecta ventana overnight: `horario_fin <= horario_inicio`
- Si un slot cae antes del `horario_inicio` (pasó medianoche), la fecha se incrementa +1 día
- Aplicado en AMBOS programadores (web + CLI)
- Buffer CLI: 15min → 30min (paridad con web)

**Commits:**
- `4b5dacd` autotok-api: fix overnight date in web programador
- `11a08bd` video_generator: fix overnight date in CLI + buffer 30min

**Archivos:** `autotok-api/api/programar.py`, `video_generator/programador.py`
